# Codex-Auftrag – Implementierungs- und Prüfauftrag für ein Canonical Workflow Framework (CWF) 1.0

## 1. Auftragstitel

**Projektweite Prüfung, kontrollierte Implementierung und Verifikation eines Canonical Workflow Framework (CWF) 1.0 für Projekt Kontinuum**

---

## 2. Ausgangslage

Projekt Kontinuum verfügt bereits über beziehungsweise plant mehrere kanonische Architektur-, Entwicklungs-, Governance-, Sicherheits- und Ausführungskomponenten.

Dazu gehören insbesondere:

* Canonical Development Framework (CDF)
* Canonical Development Governance (CDG)
* Canonical Architecture Manager (CAM)
* Canonical Authentication Framework
* Canonical Licence Management System Framework (CLMSF)
* Canonical Code Agent Framework (CCAF)
* Capability Resolution Engine (CRE)
* Execution Planner
* Orchestrator Core
* Canonical Cognitive Pipeline (CCP)
* Release Integrity
* Audit- und Provenienzstrukturen
* Artifact Lifecycle Policy
* Artifact Identity als mögliche Erweiterung
* Artifact Dependency Graph als mögliche Erweiterung

Diese Komponenten regeln unterschiedliche Aspekte von Entwicklung, Berechtigung, Planung, Ausführung, Kontrolle und Nachvollziehbarkeit.

Es fehlt jedoch bislang ein ausdrücklich definierter und technisch prüfbarer kanonischer Rahmen dafür, wie mehrstufige Arbeitsabläufe innerhalb von Projekt Kontinuum:

* beschrieben,
* versioniert,
* validiert,
* gestartet,
* ausgeführt,
* pausiert,
* fortgesetzt,
* überwacht,
* geprüft,
* abgebrochen,
* wiederholt,
* zurückgerollt,
* kompensiert,
* abgeschlossen,
* archiviert

werden.

Das geplante Canonical Workflow Framework soll diese Lücke schließen.

---

# 3. Ziel des Auftrags

Ziel dieses Auftrags ist die projektweite Prüfung, kontrollierte Implementierung und vollständige Verifikation eines:

# Canonical Workflow Framework (CWF) 1.0

Das CWF soll verbindlich definieren:

1. was innerhalb von Projekt Kontinuum als Workflow gilt,
2. wie Workflow-Definitionen eindeutig identifiziert und versioniert werden,
3. wie konkrete Workflow-Ausführungen von ihren Definitionen getrennt werden,
4. wie Schritte, Zustände und Übergänge beschrieben werden,
5. wie Rollen, Capabilities und Berechtigungen eingebunden werden,
6. wie Vorbedingungen und Nachbedingungen geprüft werden,
7. wie Freigabepunkte behandelt werden,
8. wie Fehler, Wiederholungen, Fallbacks, Rollbacks und Kompensationen geregelt werden,
9. wie Workflows pausiert und kontrolliert fortgesetzt werden,
10. wie Workflow-Ausführungen auditiert und reproduzierbar dokumentiert werden,
11. wie das CWF mit CRE, Execution Planner, Orchestrator Core, Governance, Authentication, CAM, Audit und Provenienz zusammenarbeitet,
12. wie verhindert wird, dass eine parallele Ausführungs- oder Orchestrierungsarchitektur entsteht.

Der Auftrag umfasst ausdrücklich:

* Bestandsprüfung,
* Architekturprüfung,
* Gap-Analyse,
* Definition des minimalen CWF-1.0-Scopes,
* Implementierung der kanonischen Dokumentation,
* Implementierung der Konfigurationen,
* Implementierung der JSON-Schemas,
* Implementierung eines schlanken Validators beziehungsweise einer vorhandenen Architektur entsprechenden Validierungsschicht,
* Integration in bestehende kanonische Prüfpfade, soweit dies ohne Architekturbruch möglich ist,
* Tests,
* Statusbericht,
* Abschlussbewertung.

---

# 4. Leitgrundsatz

Das CWF soll folgenden Grundsatz verbindlich verankern:

> Ein Workflow ist kein loses Skript und keine bloße Reihenfolge von Befehlen. Ein kanonischer Workflow ist ein versionierter, validierter, zustandsgebundener und nachvollziehbarer Ablauf aus eindeutig definierten Schritten, Übergängen, Bedingungen, Rollen, Capabilities, Freigaben, Eingaben, Ausgaben und Fehlerbehandlungen.

Zusätzlich gilt:

> Das CWF definiert die zulässige Struktur und das Verhalten eines Workflows. Es ersetzt weder fachliche Planung noch Capability-Auflösung noch kontrollierte Ausführung.

---

# 5. Zentrale Abgrenzung

Codex muss vor der Implementierung die Zuständigkeiten eindeutig abgrenzen.

## 5.1 Canonical Workflow Framework

Das CWF definiert:

* Workflow-Struktur,
* Workflow-Definitionen,
* Workflow-Ausführungen,
* Schritte,
* Zustände,
* Übergänge,
* Bedingungen,
* Freigaben,
* Fehlerregeln,
* Retry-Regeln,
* Pause und Fortsetzung,
* Rollback und Kompensation,
* Validierung,
* Audit-Anforderungen.

## 5.2 Capability Resolution Engine

Die CRE bestimmt:

* welche Fähigkeiten ein Workflow oder Schritt benötigt,
* welche Komponenten oder Agenten diese Fähigkeiten bereitstellen können.

Das CWF darf keine zweite Capability Resolution Engine implementieren.

## 5.3 Execution Planner

Der Execution Planner erstellt oder validiert einen konkreten Ausführungsplan.

Das CWF darf keine konkurrierende Planungsengine erzeugen.

## 5.4 Orchestrator Core

Der Orchestrator Core führt einen validierten Plan aus.

Das CWF darf keine zweite produktive Orchestrierungsengine implementieren.

## 5.5 Canonical Development Governance

CDG regelt:

* Freigabeverantwortung,
* Kontrollpflichten,
* Governance-Gates,
* Eskalation,
* unabhängige Prüfung.

Das CWF verwendet diese Regeln, ersetzt sie aber nicht.

## 5.6 Canonical Code Agent Framework

Das CCAF regelt:

* Identität,
* Rollen,
* Capabilities,
* Berechtigungen,
* Risikoklassen,
* Arbeitsgrenzen von Code-Agenten.

Das CWF beschreibt Abläufe, nicht die gesamte Governance von Code-Agenten.

## 5.7 Authentication Framework

Das Authentication Framework bestätigt:

* Akteure,
* Agenten,
* Rollen,
* Freigabeidentitäten,
* Berechtigungsnachweise.

Das CWF darf keine eigene Authentifizierungsarchitektur schaffen.

## 5.8 Canonical Artifact Manager

Der CAM verwaltet:

* kanonische Artefakte,
* Status,
* Klassifikation,
* Lifecycle,
* historische Einordnung.

Das CWF referenziert Artefakte und Ergebnisse, verwaltet sie aber nicht parallel.

## 5.9 Audit und Provenienz

Audit und Provenienz dokumentieren:

* tatsächliche Ausführungen,
* Zustandsänderungen,
* Akteure,
* Artefakte,
* Entscheidungen,
* Ergebnisse.

Das CWF definiert die benötigten Audit-Ereignisse, ersetzt bestehende Audit-Systeme jedoch nicht.

---

# 6. Verbindliche Einschränkungen

Codex darf im Rahmen dieses Auftrags nicht:

* eine zweite Orchestrierungsengine implementieren,
* den bestehenden Orchestrator Core ersetzen,
* Planungslogik aus dem Execution Planner in das CWF verschieben,
* Capability-Auflösung aus der CRE duplizieren,
* bestehende Governance umgehen,
* automatische Freigaben kritischer Workflows einführen,
* autonome Workflow-Erzeugung aktivieren,
* unbekannte Schritte dynamisch ausführen,
* nicht deklarierte Befehle starten,
* bestehende Authentifizierungs- oder Berechtigungsgrenzen lockern,
* automatische Git-Commits einführen,
* produktive Datenmigrationen durchführen,
* produktive Datenbanken unkontrolliert verändern,
* historische Workflow-Artefakte als aktive Definitionen behandeln,
* neue Top-Level-Ordner anlegen, wenn ein kanonischer Zielpfad existiert,
* eine Parallelarchitektur zu bestehenden Status-, Audit-, Planungs- oder Ausführungsstrukturen schaffen.

# 10. Minimaler Scope von CWF 1.0

CWF 1.0 soll bewusst schlank bleiben.

Der verbindlich zu prüfende und möglichst zu implementierende Mindestumfang lautet:

1. Workflow-Identität
2. Workflow-Versionierung
3. Trennung von Workflow-Definition und Workflow-Ausführung
4. Workflow-Schritte
5. Workflow-Zustände
6. zulässige Zustandsübergänge
7. Eingaben und Ausgaben
8. Preconditions und Postconditions
9. Rollenreferenzen
10. Capability-Referenzen
11. Freigabeschritte
12. Fehlerklassen
13. Retry-Regeln
14. Timeout-Regeln
15. Rollback- und Kompensationsreferenzen
16. Pause und Fortsetzung
17. Idempotenz
18. Audit- und Provenienzanforderungen
19. Validierung
20. Integrationsverträge zu CRE, Execution Planner und Orchestrator

Nicht Teil von CWF 1.0 sind:

* grafischer Workflow-Designer,
* freie visuelle Workflow-Modellierung,
* verteilte Workflow-Engine,
* autonome Workflow-Erzeugung,
* lernende Workflow-Optimierung,
* dynamische Multi-Agent-Verhandlung,
* selbstständige Prozessneugestaltung,
* Event-Streaming-Plattform,
* produktive Scheduling-Engine,
* vollständige BPMN-Implementierung.

# 11. Phase D – Begriffsdefinition

Codex soll folgende Arbeitsdefinition prüfen und kanonisch präzisieren:

> Ein kanonischer Workflow ist eine versionierte und validierbare Definition eines mehrstufigen Ablaufs aus eindeutig identifizierten Schritten, Zuständen, Übergängen, Rollen, Capability-Anforderungen, Bedingungen, Eingaben, Ausgaben, Freigaben und Fehlerregeln.

Zusätzlich ist eine Workflow-Ausführung zu definieren:

> Eine Workflow-Ausführung ist eine konkrete, eindeutig identifizierbare Instanz einer bestimmten Workflow-Definition und Workflow-Version, einschließlich ihres aktuellen Zustands, ihrer Schrittergebnisse, Freigaben, Fehler, Wiederholungen und Audit-Ereignisse.

---

# 12. Phase E – Workflow-Identität

Codex soll ein kanonisches Modell für Workflow-Definitionen implementieren beziehungsweise spezifizieren.

Mindestens zu prüfen sind:

```text
workflow_id
workflow_name
workflow_version
workflow_type
description
status
owner
risk_class
definition_hash
created_at
updated_at
valid_from
valid_until
```

Für konkrete Ausführungen mindestens:

```text
workflow_run_id
workflow_id
workflow_version
execution_plan_id
trigger_type
trigger_reference
initiated_by
current_state
current_step_id
started_at
paused_at
resumed_at
completed_at
final_status
```

## 12.1 Anforderungen

* `workflow_id` muss dauerhaft stabil sein.
* `workflow_version` muss eindeutig sein.
* Eine Workflow-Ausführung muss an eine konkrete Definition und Version gebunden bleiben.
* Änderungen an einer Workflow-Definition dürfen laufende Ausführungen nicht stillschweigend verändern.
* Der Hash der verwendeten Definition soll prüfbar sein.
* Historische Workflow-Versionen dürfen nicht automatisch aktiv werden.

---

# 13. Phase F – Workflow-Typen

CWF 1.0 soll mindestens folgende Typen prüfen und gegebenenfalls unterstützen:

## 13.1 Sequential Workflow

```text
A → B → C
```

## 13.2 Conditional Workflow

```text
A → Bedingung
     ├─ erfüllt → B
     └─ nicht erfüllt → C
```

## 13.3 Approval Workflow

```text
Analyse → Planung → Freigabe → Ausführung
```

## 13.4 Review Workflow

```text
Ergebnis → unabhängige Prüfung → Annahme oder Rückgabe
```

## 13.5 Recovery Workflow

```text
Fehler → Diagnose → Rollback oder Kompensation → Abschluss
```

## 13.6 Parallel Workflow

Nur dann in CWF 1.0 implementieren, wenn:

* bestehende Architektur dies bereits unterstützt,
* Synchronisationsbedingungen eindeutig modellierbar sind,
* keine neue konkurrierende Runtime erforderlich ist.

Andernfalls ist Parallelität nur als spätere Erweiterung zu dokumentieren.

# 15. Phase H – Workflow-Zustände

Codex soll die vorhandenen projektweiten Statuswerte prüfen.

Nur wenn keine geeigneten kanonischen Zustände bestehen, ist ein schlanker CWF-Statussatz einzuführen.

Ausgangsvorschlag:

```text
DRAFT
VALIDATING
READY
WAITING
RUNNING
PAUSED
BLOCKED
RETRYING
ROLLING_BACK
FAILED
ABORTED
COMPLETED
VERIFIED
REJECTED
ARCHIVED
```

## 15.1 Anforderungen

* Zustände müssen kontrollierte Enumerationen sein.
* Statusübergänge müssen validierbar sein.
* `VERIFIED` darf nicht ohne vorherigen Abschluss erreicht werden.
* `ARCHIVED` darf keine aktive Ausführung darstellen.
* `FAILED`, `ABORTED` und `REJECTED` müssen unterscheidbar sein.
* `PAUSED` muss fortsetzbar oder kontrolliert abbrechbar sein.
* `BLOCKED` muss eine Blockadeursache besitzen.

# 16. Phase I – Übergangsmodell

Jeder Übergang muss mindestens beschreiben:

```text
transition_id
from_state
to_state
condition
authorized_roles
required_approvals
required_capabilities
audit_event
```

## 16.1 Beispiel

```text
from_state: READY
to_state: RUNNING
condition: execution_plan_validated == true
authorized_roles:
  - orchestrator
required_approvals:
  - governance_gate
```

## 16.2 Verbote

Nicht zulässig sind:

* freie Statusänderungen,
* Übergänge ohne Ausgangszustand,
* Übergänge ohne Zielzustand,
* Umgehung von Freigaben,
* Übergang zu `VERIFIED` ohne Prüfnachweis,
* Übergang aus `ARCHIVED` in einen aktiven Zustand ohne neue Instanz,
* stiller Wechsel der Workflow-Version während einer Ausführung.

# 18. Phase K – Eingaben und Ausgaben

Jeder Workflow und jeder Schritt muss deklarierte Ein- und Ausgaben besitzen.

Mindestens:

```text
input_id
input_type
schema_reference
required
source
```

```text
output_id
output_type
schema_reference
artifact_reference
required
```

## 18.1 Schutzregel

Ein Workflow darf keine nicht deklarierten produktiven Seiteneffekte erzeugen.

Seiteneffekte müssen:

* als Ergebnis,
* als Artefakt,
* als Statusänderung,
* als Audit-Ereignis,
* als Folgeauftrag

erkennbar sein.

# 19. Phase L – Preconditions und Postconditions

Codex soll ein deterministisches Modell für Vor- und Nachbedingungen definieren.

## 19.1 Mögliche Preconditions

* gültiger Auftrag vorhanden,
* Workflow-Definition gültig,
* Workflow-Version aktiv,
* Akteur authentifiziert,
* erforderliche Capabilities auflösbar,
* erforderliche Artefakte vorhanden,
* Rückfallpunkt vorhanden,
* benötigte Freigaben vorhanden.

## 19.2 Mögliche Postconditions

* erwartete Ergebnisse vorhanden,
* Artefakte registriert,
* Tests ausgeführt,
* Status korrekt,
* Audit vollständig,
* Abschlussbericht erzeugt.

## 19.3 Einschränkung

Bedingungen dürfen nicht als unkontrollierter frei ausführbarer Code gespeichert werden.

Zu bevorzugen sind:

* kontrollierte Enumerationen,
* Referenzen,
* bekannte Validatoren,
* deklarative Ausdrücke mit klar begrenzter Semantik.

---

# 20. Phase M – Rollen und Capabilities

Das CWF darf Rollen und Capabilities nur referenzieren.

Zu prüfen sind mindestens:

```text
workflow_owner
planner
executor
reviewer
approver
auditor

```

Capability-Beispiele:

```text

workflow.pause
workflow.resume

workflow.retry
workflow.rollback

```

Es darf keine parallele Capability-Registry angelegt werden, wenn eine geeignete kanonische Registry existiert.

# 21. Phase N – Freigabepunkte

CWF 1.0 soll kontrollierte Gates unterstützen.

Mindestens zu prüfen:

```text
APPROVAL_GATE
REVIEW_GATE
SECURITY_GATE
RELEASE_GATE
HUMAN_CONFIRMATION_GATE
```

Jeder Freigabepunkt muss enthalten:

```text
gate_id
gate_type
required_role
required_actor_count
approval_status
approved_by
approved_at
rejection_reason
```

## 21.1 Grundsatz

Ein Freigabepunkt darf:

* nicht automatisch übersprungen,
* nicht durch den ausführenden Schritt selbst stillschweigend erfüllt,
* nicht nachträglich fingiert

werden.

---

# 22. Phase O – Fehlerbehandlung

Codex soll ein kanonisches Fehlerbehandlungsmodell definieren.

Mindestens:

```text
STOP
RETRY
SKIP
FALLBACK
ROLLBACK
COMPENSATE
ESCALATE
ABORT
```

## 22.1 Anforderungen

Jeder kritische Schritt muss angeben:

* welche Fehler erwartet werden,
* welche Fehler retryfähig sind,
* welche Fehler nicht retryfähig sind,
* wann abgebrochen wird,
* ob Rollback möglich ist,
* ob Kompensation erforderlich ist,
* wer eskaliert wird.

## 22.2 Schutzregel

Ohne definierte Fehlerstrategie muss ein kritischer Fehler den Workflow anhalten.

---

# 23. Phase P – Retry-Regeln

Wiederholungen müssen begrenzt und auditierbar sein.

Mindestens:

```text
max_attempts
delay_seconds
backoff_strategy
retryable_errors
non_retryable_errors
reset_timeout_on_retry
```

## 23.1 Verbote

* keine unbegrenzten Wiederholungen,
* keine Wiederholung irreversibler Schritte ohne Idempotenz- oder Kompensationsregel,
* keine automatische Wiederholung nach Authentifizierungs- oder Berechtigungsfehlern,
* keine Wiederholung bei strukturell ungültigem Workflow.

---

# 24. Phase Q – Timeout-Regeln

Für lang laufende oder blockierbare Schritte sind Timeout-Regeln vorzusehen.

Mindestens:

```text
timeout_seconds
on_timeout
timeout_error_code
escalation_required
```

Mögliche Reaktionen:

```text
FAIL
PAUSE
RETRY
ESCALATE
ABORT
```

---

# 25. Phase R – Rollback und Kompensation

Codex soll klar zwischen Rollback und Kompensation unterscheiden.

## 25.1 Rollback

Rollback stellt einen früheren technischen Zustand wieder her.

Beispiele:

* Dateiänderung zurücksetzen,
* Konfiguration wiederherstellen,
* Datenbanktransaktion rückgängig machen.

## 25.2 Kompensation

Kompensation führt eine ausgleichende Aktion aus, wenn ein echter Rollback nicht möglich ist.

Beispiele:

* Korrekturmeldung,
* Gegenbuchung,
* neues Ersatzartefakt,
* Deaktivierung eines bereits veröffentlichten Ergebnisses.

## 25.3 Anforderungen

Ein Schritt soll angeben:

```text
rollback_supported
rollback_reference
compensation_supported
compensation_reference
irreversible
```

Irreversible Schritte müssen besonders gekennzeichnet und freigegeben werden.

# 26. Phase S – Idempotenz

Jeder Schritt soll angeben, ob er gefahrlos mehrfach ausgeführt werden kann.

```text
idempotent: true | false
```

Für nicht idempotente Schritte ist zu prüfen:

* dedizierter Idempotency Key,
* Wiederholungsverbot,
* Kompensationsregel,
* besondere Freigabe.

# 27. Phase T – Pause und Fortsetzung

CWF 1.0 soll eine kontrollierte Pause und Fortsetzung unterstützen.

Bei einer Pause müssen mindestens gespeichert werden:

```text

current_step
pending_steps
inputs
outputs
approvals
definition_hash
paused_at
pause_reason
```

Vor Fortsetzung muss erneut geprüft werden:

* Workflow-Version unverändert,
* Definition-Hash gültig,
* Projektzustand kompatibel,
* erforderliche Artefakte vorhanden,
* Berechtigungen weiterhin gültig,
* Freigaben nicht abgelaufen,
* Rückfallpunkt weiterhin nutzbar,
* keine auftragsfremden Konflikte entstanden.

Eine Fortsetzung darf nicht blind am letzten Befehl ansetzen.

# 28. Phase U – Workflow-Validierung

* genau ein definierter Startpunkt,
* mindestens ein Endzustand,
* keine ungültigen Zustände,

* keine unerreichbaren Pflichtschritte,
* keine unkontrollierten Zyklen,
* bekannte Schrittarten,
* bekannte Capability-Referenzen,

# 30. Phase W – Integrationsverträge

Codex soll dokumentierte Integrationsverträge erstellen.

## 30.1 CRE

Das CWF liefert:

* benötigte Capabilities pro Workflow und Schritt.

Die CRE liefert:

* Auflösungsresultat,
* verfügbare Provider,
* erfüllte oder fehlende Capabilities.

## 30.2 Execution Planner

Das CWF liefert:

* gültige Workflow-Struktur,
* Schrittbeziehungen,
* Bedingungen,
* Constraints,
* Freigabepunkte.

Der Execution Planner liefert:

* konkreten validierten Ausführungsplan.

## 30.3 Orchestrator Core

Der Orchestrator erhält:

* validierten Plan,
* zulässige Schritte,
* Statusregeln,
* Freigaben,
* Ausführungsgrenzen.

Der Orchestrator darf:

* validierte Schritte ausführen,
* Zustände melden,
* Fehler melden.

Der Orchestrator darf nicht:

* Workflow-Regeln eigenmächtig ändern,
* Freigaben erzeugen,
* neue Schritte erfinden,
* Planungsentscheidungen übernehmen.

## 30.4 CDG

CDG bestimmt:

* notwendige Prüfungen,
* Freigabestufen,
* Rollentrennung,
* Eskalationsregeln.

## 30.5 CCAF

CCAF bestimmt:

* welche Agentenrollen Schritte übernehmen dürfen,
* welche Permissions gelten,
* welche Risikoklassen besondere Prüfungen erfordern.

## 30.6 Authentication

Authentication bestätigt:

* Identität des Auslösers,
* Identität des Ausführenden,
* Identität des Freigebenden.

## 30.7 CAM

Der CAM registriert:

* Workflow-Definitionen,
* Workflow-Konfigurationen,
* erzeugte Berichte,
* Ergebnisartefakte.

## 30.8 Audit und Provenienz

Audit und Provenienz speichern:

* Zustandswechsel,
* Ausführungsschritte,
* Ergebnisse,
* Entscheidungen,
* Fehler,
* Freigaben.

# 32. Phase Y – Technische Implementierung

## 32.1 Erwartete technische Kernfunktion

## 32.2 Keine Runtime-Engine

Die Implementierung darf keine neue vollständige Workflow-Runtime oder Orchestrierungsengine schaffen.

* alle Tests vorhanden sind.

Andernfalls ist die Runtime-Integration als Folgeauftrag zu dokumentieren.

# 33. Phase Z – Teststrategie

Codex muss automatisierte Tests implementieren.

## 33.1 Schema-Tests

* gültige Workflow-Definition wird akzeptiert,
* fehlende Workflow-ID wird abgelehnt,
* ungültige Version wird abgelehnt,
* doppelte Schritt-ID wird abgelehnt,
* unbekannter Zustand wird abgelehnt,
* unbekannte Schrittart wird abgelehnt.

## 33.2 Zustands- und Übergangstests

* gültiger Übergang wird akzeptiert,
* ungültiger Übergang wird abgelehnt,
* `DRAFT → VERIFIED` wird abgelehnt,
* `READY → RUNNING` ohne Planfreigabe wird abgelehnt,
* archivierte Ausführung kann nicht direkt fortgesetzt werden.

## 33.3 Strukturtests

* fehlender Startpunkt wird erkannt,
* mehrere unzulässige Startpunkte werden erkannt,
* fehlender Endpunkt wird erkannt,
* unerreichbarer Pflichtschritt wird erkannt,
* unkontrollierte Schleife wird erkannt.

## 33.4 Capability-Tests

* bekannte Capability wird akzeptiert,
* unbekannte Capability wird erkannt,
* fehlende Capability wird im Validierungsresultat ausgewiesen,
* keine parallele Capability-Registry wird erzeugt.

## 33.5 Approval-Tests

* notwendiges Gate vorhanden,
* fehlendes Gate wird erkannt,
* Selbstfreigabe bei kritischem Schritt wird abgelehnt,
* abgelehnte Freigabe stoppt Übergang.

## 33.6 Retry-Tests

* gültige begrenzte Retry-Policy,
* negative Versuchszahl wird abgelehnt,
* unbegrenzte Wiederholung wird abgelehnt,
* nicht idempotenter Schritt ohne Schutz wird beanstandet.

## 33.7 Rollback- und Kompensationstests

* gültige Rollback-Referenz,
* fehlende Referenz bei kritischem reversiblem Schritt,
* irreversibler Schritt ohne besondere Kennzeichnung,
* Kompensationsreferenz bei nicht rückrollbarem Schritt.

## 33.8 Pause-und-Fortsetzung-Tests

* Pause speichert Zustand,
* Fortsetzung mit unverändertem Hash möglich,
* Fortsetzung mit geänderter Definition wird abgelehnt,
* abgelaufene Freigabe blockiert Fortsetzung.

## 33.9 Audit-Tests

* Zustandswechsel erzeugt Audit-Ereignis,
* Fehler erzeugt Audit-Ereignis,
* Retry erzeugt Audit-Ereignis,
* Freigabe erzeugt Audit-Ereignis,
* Abschluss ist mit Workflow-Run verknüpft.

## 33.10 Integrationstests

Soweit vorhandene Architektur dies ermöglicht:

* CRE
* Execution Planner
* Orchestrator Core
* CAM
* Audit
* Provenienz
* Release Integrity

# 37. Sicherheitsregeln

CWF 1.0 muss mindestens folgende Regeln verbindlich dokumentieren und soweit möglich validieren:

* keine unbekannten Schrittarten,
* keine nicht deklarierten produktiven Seiteneffekte,
* keine unkontrollierten Schleifen,
* keine unbegrenzten Retries,
* keine Umgehung von Freigabepunkten,
* keine eigenmächtige Scope-Erweiterung,
* keine Änderung laufender Workflow-Definitionen,
* keine unbekannten Capabilities,
* keine automatische Selbstfreigabe kritischer Abläufe,
* keine Fortsetzung nach kritischem Fehler ohne definierte Regel,
* keine Ausführung ohne gültige Definition,
* keine Ausführung ohne gültige Version,
* keine Ausführung ohne notwendige Identität und Berechtigung,
* keine stillen Statusänderungen,
* keine unprotokollierten Rollbacks,
* keine unprotokollierten Kompensationen.

---

# 38. Abbruchbedingungen

Codex muss die Implementierung abbrechen oder auf Dokumentation und Prüfbericht begrenzen, wenn:

* bereits ein gleichwertiges aktives Workflow Framework existiert,
* CWF und Execution Planner nicht sauber trennbar sind,
* CWF und Orchestrator nicht sauber trennbar sind,
* ein Rückfallpunkt nicht bestimmbar ist,
* fremde uncommittete Änderungen gefährdet wären,
* zentrale Statusdefinitionen widersprüchlich sind,
* Capability-Registries mehrfach und widersprüchlich existieren,
* kanonische Zielpfade nicht bestimmbar sind,
* eine Implementierung nur durch Änderung produktiver Runtime-Kernpfade möglich wäre,
* bestehende Tests bereits vor dem Auftrag fehlschlagen und keine klare Abgrenzung möglich ist,
* die Implementierung eine neue unkontrollierte Workflow-Engine erzeugen würde.

In diesem Fall ist kein improvisierter Ersatz zu implementieren.

# 40. Erfolgskriterien

Der Auftrag gilt nur dann als erfolgreich abgeschlossen, wenn:

* die vorhandene Architektur zuerst geprüft wurde,
* CWF klar von Planner und Orchestrator abgegrenzt ist,
* keine Parallelarchitektur entstanden ist,
* Workflow-Definition und Workflow-Ausführung getrennt sind,
* Zustände und Übergänge maschinenlesbar definiert sind,
* Schritte validierbar sind,
* Fehler-, Retry-, Timeout-, Rollback- und Kompensationsregeln definiert sind,
* Freigabepunkte nicht umgangen werden können,
* Pause und Fortsetzung kontrolliert modelliert sind,
* Audit- und Provenienzanforderungen enthalten sind,
* Schemas vorhanden und getestet sind,
* ein Validator oder eine gleichwertige bestehende Prüfschicht vorhanden ist,
* Tests erfolgreich ausgeführt wurden,
* Dokumentation, Konfigurationen und Statusbericht konsistent sind,
* der Abschlussbericht alle tatsächlichen Änderungen offenlegt.

---

# 41. Nicht zulässige Ergebnisse

Der Auftrag ist nicht erfüllt, wenn:

* lediglich ein allgemeines Workflow-Dokument erstellt wird,
* nur JSON-Dateien ohne Validator entstehen,
* ein Validator ohne Tests entsteht,
* eine zweite Orchestrierungsengine implementiert wird,
* Workflow-Zustände frei geändert werden können,
* Definition und Ausführung vermischt werden,
* Retry ohne Begrenzung möglich ist,
* Rollback und Kompensation nicht unterschieden werden,
* laufende Workflows unkontrolliert auf neue Versionen wechseln,
* Freigaben nur als unverbindlicher Text behandelt werden,
* Audit-Ereignisse fehlen,
* bestehende kanonische Registries dupliziert werden,
* unbekannte Capabilities erlaubt werden,
* Parallelität eingeführt wird, ohne Synchronisation und Fehlerbehandlung zu definieren.
