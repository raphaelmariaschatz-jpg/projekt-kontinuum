# Codex-Auftrag – Prüfung und Vorbereitung eines Canonical Code Agent Framework (CCAF) 1.0

## 1. Auftragstitel

**Prüfung, Architekturdefinition und kontrollierte Vorbereitung eines Canonical Code Agent Framework (CCAF) 1.0 für Projekt Kontinuum**

---

## 2. Ausgangslage

Projekt Kontinuum verfügt bereits über beziehungsweise plant mehrere kanonische Architektur-, Entwicklungs-, Governance-, Sicherheits- und Betriebsbausteine.

Dazu gehören insbesondere:

* Canonical Development Framework (CDF)
* Canonical Development Governance (CDG)
* Canonical Authentication Framework
* Canonical Licence Management System Framework (CLMSF)
* Canonical Architecture Manager (CAM)
* Capability Resolution Engine (CRE)
* Execution Planner
* Orchestrator Core
* Canonical Cognitive Pipeline (CCP)
* Release Integrity
* Audit-, Provenienz- und Lifecycle-Strukturen
* Artifact Identity und Artifact Dependency Graph als mögliche Erweiterungen

Mit zunehmender Nutzung automatisierter Entwicklungswerkzeuge entsteht die Notwendigkeit, verbindlich festzulegen, wie Code-Agenten innerhalb von Projekt Kontinuum identifiziert, beauftragt, berechtigt, kontrolliert, geprüft und protokolliert werden.

Ein Code-Agent darf dabei nicht allein anhand seiner technischen Fähigkeiten handeln. Sein zulässiges Verhalten muss sich aus einem kanonischen Auftrag, einer registrierten Rolle, einem Berechtigungsprofil, einer Risikoklasse und den geltenden Governance-Regeln ergeben.

Das geplante Framework soll nicht auf Codex als einzelnes Produkt beschränkt sein. Es soll als allgemeiner, hersteller- und modellunabhängiger Rahmen für heutige und zukünftige Code-Agenten dienen.

Dazu können später beispielsweise gehören:

* Codex
* lokale Code-Modelle
* spezialisierte Prüfagenten
* Testagenten
* Dokumentationsagenten
* Sicherheitsagenten
* Migrationsagenten
* Release-Agenten
* zukünftige interne Entwicklungsagenten von K

---

## 3. Ziel des Auftrags

Ziel dieses Auftrags ist die gründliche Prüfung und konzeptionelle Vorbereitung eines:

**Canonical Code Agent Framework (CCAF) 1.0**

Das CCAF soll verbindlich definieren:

1. was innerhalb von Projekt Kontinuum als Code-Agent gilt,
2. wie Code-Agenten eindeutig identifiziert werden,
3. welche Rollen und Fähigkeiten sie besitzen können,
4. wie ihre Berechtigungen begrenzt werden,
5. wie ein kanonischer Agentenauftrag aufgebaut ist,
6. wie Analyse, Planung, Ausführung, Prüfung und Berichterstattung voneinander getrennt werden,
7. wie Risiken klassifiziert werden,
8. wie Agentenaktionen protokolliert und nachvollziehbar gemacht werden,
9. wie das CCAF mit bestehenden kanonischen Komponenten zusammenwirkt,
10. wie spätere kontrollierte oder teilweise autonome Codearbeit vorbereitet werden kann, ohne sie mit diesem Auftrag bereits zu aktivieren.

Der Auftrag dient zunächst der:

* Bestandsprüfung,
* Architekturprüfung,
* Abgrenzung,
* Spezifikation,
* Dokumentationsvorbereitung,
* Schema- und Artefaktplanung.

Eine produktive Aktivierung autonomer Code-Agenten ist ausdrücklich nicht Gegenstand dieses Auftrags.

---

## 4. Grundsatz

Das CCAF soll folgenden Grundsatz verbindlich verankern:

> Kein Code-Agent darf allein aufgrund seiner technischen Möglichkeiten handeln. Jede Handlung muss durch eine eindeutige Agentenidentität, einen kanonischen Auftrag, eine zulässige Rolle, ein gültiges Berechtigungsprofil, eine Risikoklasse und eine überprüfbare Governance-Freigabe gedeckt sein.

Zusätzlich gilt:

> Ein Code-Agent darf seinen eigenen Arbeitsumfang nicht eigenmächtig erweitern und seine eigene Arbeit nicht allein endgültig freigeben.

---

## 5. Wichtige Einschränkung dieses Auftrags

Dieser Auftrag ist zunächst ein:

**Prüf-, Definitions- und Vorbereitungsauftrag.**

Codex darf daher nicht ohne eindeutige Freigabe:

* produktive Agentenlaufzeiten aktivieren,
* autonome Schreibrechte einführen,
* bestehende Entwicklungsabläufe ersetzen,
* produktive Orchestrierungswege verändern,
* Berechtigungen bestehender Komponenten erweitern,
* bestehende Sicherheitsgrenzen lockern,
* Git-Automatisierungen aktivieren,
* automatische Commits einführen,
* automatische Löschvorgänge ermöglichen,
* produktive Datenbanken verändern,
* Agenten selbstständig Entscheidungen über Freigaben treffen lassen,
* bestehende Governance-Regeln umgehen oder neu interpretieren.

Zunächst ist zu prüfen, zu dokumentieren und kontrolliert vorzubereiten.

# 6. Phase A – Kontrollierter Einstieg

Vor jeder fachlichen Ausarbeitung muss Codex den aktuellen Projektzustand kontrolliert erfassen.

## 6.1 Rückfallpunkt

Codex muss zu Beginn:

* den aktuellen Git-Status prüfen,
* den aktuellen Branch benennen,
* den aktuellen Commit-Hash dokumentieren,
* vorhandene uncommittete Änderungen erfassen,
* bestehende Staged-Änderungen gesondert ausweisen,
* prüfen, ob bereits ein geeigneter Rückfallpunkt vorhanden ist.

Codex darf keine bestehenden Änderungen überschreiben, zurücksetzen, bereinigen oder stillschweigend einbeziehen.

Falls der Arbeitsbaum nicht sauber ist, muss Codex:

* den Zustand dokumentieren,
* betroffene Dateien nennen,
* zwischen auftragsrelevanten und auftragsfremden Änderungen unterscheiden,
* ohne ausdrückliche Freigabe keine fremden Änderungen verändern.

# 8. Phase C – Begriffsdefinition und Abgrenzung

Codex soll eine präzise Definition für einen Code-Agenten erarbeiten.

## 8.1 Arbeitsdefinition

Zu prüfen ist folgende Ausgangsdefinition:

> Ein Code-Agent ist eine identifizierbare, regelgebundene Software- oder Modellinstanz, die innerhalb eines ausdrücklich freigegebenen Auftrags Quellcode, Konfigurationen, Tests, Dokumentation oder zugehörige Entwicklungsartefakte analysiert, plant, erzeugt, verändert, prüft oder verwaltet.

Codex soll prüfen, ob diese Definition:

* vollständig,
* ausreichend eng,
* technisch umsetzbar,
* modellunabhängig,
* mit bestehenden Projektbegriffen kompatibel ist.

## 8.2 Abgrenzung

Das CCAF muss klar abgegrenzt werden von:

### Canonical Development Framework

Das CDF regelt den übergeordneten Entwicklungsprozess.

### Canonical Development Governance

CDG regelt Verantwortlichkeiten, Freigaben, Kontrollregeln und Governance.

### Canonical Authentication Framework

Das Authentication Framework bestätigt Identitäten und Berechtigungsnachweise.

### Canonical Licence Management System Framework

Das CLMSF regelt Lizenzierung, Editionen, Rechte und zulässige Nutzung.

### Capability Resolution Engine

Die CRE bestimmt, welche Fähigkeiten für einen Auftrag benötigt werden und welche Komponente sie bereitstellen kann.

### Execution Planner

Der Execution Planner erstellt oder validiert einen kontrollierten Ausführungsplan.

### Orchestrator Core

Der Orchestrator führt validierte Pläne aus, soll aber keine eigenständige Entwicklungsentscheidung übernehmen.

### Canonical Cognitive Pipeline

Die CCP regelt kognitive oder verarbeitungsbezogene Abläufe von K und darf nicht mit Code-Agenten-Governance vermischt werden.

### Canonical Artifact Manager

Der CAM verwaltet kanonische Artefakte, deren Status, Klassifikation und Lifecycle.

Das CCAF soll diese Systeme nicht duplizieren, sondern ihre Anforderungen für Code-Agenten verbindlich zusammenführen.

# 9. Phase D – Zielarchitektur des CCAF 1.0

Codex soll eine schlanke, klar abgegrenzte Zielarchitektur entwerfen.

## 9.1 Kernbereiche

Das CCAF sollte mindestens folgende Bereiche definieren:

1. Agent Identity
2. Agent Registry
3. Agent Roles
4. Agent Capabilities
5. Agent Permission Profiles
6. Canonical Agent Task
7. Agent Operating Modes
8. Risk Classification
9. Approval and Control Gates
10. Execution Boundaries
11. Verification Requirements
12. Audit and Provenance
13. Agent Reporting
14. Failure and Abort Rules
15. Integration Contracts
16. Future Autonomy Boundaries

## 9.2 Architekturprinzip

Codex soll prüfen, ob folgende Verarbeitungskette geeignet ist:

```text
Canonical Agent Task
        ↓
Agent Identity Validation
        ↓
Role and Permission Resolution
        ↓
Capability Resolution
        ↓
Risk Classification
        ↓
Governance Gate
        ↓

        ↓

        ↓

        ↓

```

Dabei muss klar bleiben:

* CCAF definiert Regeln und Verträge.
* CRE löst Fähigkeiten auf.
* Execution Planner plant.
* Orchestrator führt aus.
* Governance entscheidet über Freigabebedingungen.
* Authentication bestätigt Identität.
* Audit und Provenienz dokumentieren die Ausführung.
* Raphael beziehungsweise eine ausdrücklich autorisierte Instanz behält die endgültige Kontrolle.

# 11. Phase F – Agentenrollen

Codex soll ein kanonisches Rollenmodell definieren.

Mindestens zu prüfen sind:

## 11.1 Analysis Agent

Darf:

* lesen,
* analysieren,
* Risiken benennen,
* Berichte erstellen.

Darf nicht:

* produktive Dateien verändern,
* Dateien löschen,
* Freigaben erteilen.

## 11.2 Planning Agent

Darf:

* Änderungspläne,
* Migrationspläne,
* Testpläne,
* betroffene Artefakte,
* Risiken und Rückfallstrategien definieren.

Darf nicht ohne gesonderte Freigabe:

* den Plan selbst produktiv ausführen.

## 11.3 Implementation Agent

Darf ausschließlich innerhalb eines freigegebenen Scopes:

* Dateien erstellen,
* Dateien ändern,
* genehmigte Implementierungsschritte ausführen.

## 11.4 Review Agent

Prüft:

* Architekturkonformität,
* Scope-Einhaltung,
* Seiteneffekte,
* Qualität,
* mögliche Regressionen.

## 11.5 Test Agent

Darf:

* Tests identifizieren,
* Tests ausführen,
* Testergebnisse dokumentieren,
* fehlende Tests benennen.

## 11.6 Security Agent

Prüft insbesondere:

* Authentifizierung,
* Autorisierung,
* Geheimnisse,
* Eingabevalidierung,
* Datei- und Prozesszugriffe,
* Dependency-Risiken,
* Injection-Risiken,
* Privilegienausweitung.

## 11.7 Migration Agent

Darf nur mit besonderer Freigabe:

* Dateien verschieben,
* Strukturen migrieren,
* Versionsstände überführen,
* Referenzen aktualisieren.

## 11.8 Documentation Agent

Darf:

* Dokumentation,
* Glossar,
* Statusberichte,
* Chronik,
* Architekturübersichten aktualisieren.

## 11.9 Release Agent

Prüft:

* Release Integrity,
* Manifest-Vollständigkeit,
* Tests,
* Dokumentation,
* Statusberichte,
* Freigabereife.

Codex soll prüfen, welche Rollen wirklich für Version 1.0 notwendig sind und welche zunächst nur als Erweiterung vorgesehen werden sollten.

# 15. Phase J – Kanonischer Arbeitszyklus

Codex soll einen verbindlichen Code-Agenten-Arbeitszyklus definieren.

Zu prüfen ist folgende Struktur:

## Phase 1 – Identify

Der Agent identifiziert:

* sich selbst,
* den Auftrag,
* seine Rolle,
* seine Berechtigungen,
* die Risikoklasse,
* den Projektzustand,
* den Rückfallpunkt.

## Phase 2 – Inspect

Der Agent prüft:

* relevante Dateien,
* Abhängigkeiten,
* aktive und historische Artefakte,
* Tests,
* Konfigurationen,
* Governance-Regeln,
* mögliche Konflikte.

## Phase 3 – Plan

Der Agent erstellt:

* geplante Schritte,
* betroffene Artefakte,
* erwartete Änderungen,
* Risiken,
* Testplan,
* Rückfallstrategie.

## Phase 4 – Authorize

Vor jeder schreibenden Aktion wird geprüft:

* Ist die Aktion durch den Auftrag gedeckt?
* Ist sie durch die Rolle erlaubt?
* Ist die Capability registriert?
* Ist die Permission vorhanden?
* Liegt der Zielpfad im Scope?
* Ist die Risikoklasse berücksichtigt?
* Ist eine zusätzliche Freigabe erforderlich?

## Phase 5 – Execute

Nur freigegebene Schritte dürfen umgesetzt werden.

## Phase 6 – Verify

Es sind durchzuführen:

* Tests,
* Scope-Prüfung,
* Diff-Prüfung,
* Architekturprüfung,
* Integritätsprüfung,
* Prüfung unerwarteter Änderungen.

## Phase 7 – Report

Der Agent erstellt einen nachvollziehbaren Abschlussbericht.

Codex soll prüfen, wie dieser Ablauf mit:

**Erkennen – Schaffen – Vollenden**

in Übereinstimmung gebracht werden kann, ohne technische Prüfschritte zu vereinfachen oder auszulassen.

# 16. Phase K – Betriebsmodi

Codex soll folgende Betriebsmodi prüfen:

## 16.1 Read-only Mode

* nur lesen,
* analysieren,
* berichten.

## 16.2 Proposal Mode

* analysieren,
* planen,
* konkrete Änderungsvorschläge erstellen,
* keine produktive Umsetzung.

## 16.3 Controlled Write Mode

* begrenzte Änderungen,
* nur im freigegebenen Scope,
* verpflichtende Tests und Diff-Prüfung.

## 16.4 Migration Mode

* kontrollierte Verschiebungen,
* Referenzanpassungen,
* vollständige Rückfallstrategie,
* besondere Freigabe.

## 16.5 Emergency Repair Mode

* nur für klar definierte Störungen,
* minimaler Eingriff,
* keine Architekturweiterentwicklung,
* vollständige Nachprüfung.

## 16.6 Autonomous Maintenance Mode

Dieser Modus darf im Rahmen dieses Auftrags ausschließlich konzeptionell betrachtet werden.

Er darf nicht aktiviert oder produktiv vorbereitet werden, sofern dies Schreibrechte oder eigenständige Entscheidungen ermöglicht.

Codex soll für Version 1.0 empfehlen, welche Modi tatsächlich enthalten sein sollen.

---

# 17. Phase L – Risikoklassen

Codex soll ein kanonisches Risikomodell prüfen.

Ausgangsvorschlag:

## CCAF-R0 – Analyse

* keine Änderungen,
* keine produktiven Befehle.

## CCAF-R1 – Geringes Risiko

* Dokumentation,
* isolierte Ergänzungen,
* keine kritischen Abhängigkeiten.

## CCAF-R2 – Begrenzte Codeänderung

* klar begrenzter Scope,
* lokale Tests erforderlich.

## CCAF-R3 – Erhöhtes Risiko

* mehrere Komponenten,
* öffentliche Schnittstellen,
* zentrale Konfiguration,
* Datenmodell oder Architektur betroffen.

## CCAF-R4 – Kritische Änderung

* Authentifizierung,
* Autorisierung,
* Lizenzierung,
* Migration,
* Release,
* Datenbank,
* Orchestrierung,
* zentrale Governance.

## CCAF-R5 – Systemweite oder irreversible Änderung

* weitreichende Migration,
* Löschung,
* produktive Datenumwandlung,
* zentrale Sicherheitsarchitektur,
* nicht einfach rückrollbare Eingriffe.

Codex soll prüfen:

* ob diese Klassen mit bestehenden Risikoklassen kollidieren,
* ob eine eigene CCAF-Kennung notwendig ist,
* welche Freigaben und Prüfungen je Risikoklasse erforderlich sind.

# 18. Phase M – Kontroll- und Freigabepunkte

Codex soll verbindliche Gates definieren.

Mindestens zu prüfen sind:

## Gate 1 – Identity Gate

Ist der Agent eindeutig identifiziert?

## Gate 2 – Task Gate

Ist ein gültiger kanonischer Auftrag vorhanden?

## Gate 3 – Role Gate

Ist die aktive Rolle für den Auftrag zulässig?

## Gate 4 – Capability Gate

Besitzt der Agent die benötigten Fähigkeiten?

## Gate 5 – Permission Gate

Darf der Agent diese Fähigkeiten im konkreten Scope einsetzen?

## Gate 6 – Risk Gate

Sind alle Anforderungen der Risikoklasse erfüllt?

## Gate 7 – Plan Gate

Liegt ein validierter Ausführungsplan vor?

## Gate 8 – Execution Gate

Sind die konkreten Aktionen freigegeben?

## Gate 9 – Verification Gate

Wurden Tests, Integrität und Scope-Einhaltung geprüft?

## Gate 10 – Final Approval Gate

Wurde die Arbeit von einer unabhängigen oder ausdrücklich autorisierten Instanz freigegeben?

Ein Agent darf fehlgeschlagene Gates nicht selbstständig umgehen.

# 19. Phase N – Vier-Augen-Prinzip und Rollentrennung

Codex soll prüfen, wie eine unabhängige Kontrolle umgesetzt werden kann.

Für kritische Aufgaben ist folgende Trennung zu bewerten:

```text
Analysis Agent
        ↓
Planning Agent
        ↓
Implementation Agent
        ↓
Review / Test / Security Agent
        ↓
Finale autorisierte Freigabe
```

Zu definieren ist:

* wann Rollen kombiniert werden dürfen,
* wann eine Trennung verpflichtend ist,
* wann ein Agent seine eigene Arbeit prüfen darf,
* wann eine unabhängige Prüfung erforderlich ist,
* wann Raphael persönlich freigeben muss,
* wie dies technisch und dokumentarisch nachgewiesen wird.

Grundsatz:

> Selbstprüfung darf eine unabhängige Prüfung ergänzen, aber bei kritischen Änderungen nicht ersetzen.

# 20. Phase O – Verbote und Schutzregeln

Das CCAF muss mindestens folgende Verbote prüfen und gegebenenfalls verbindlich festschreiben:

* keine eigenmächtige Scope-Erweiterung,
* keine stillen Architekturänderungen,
* keine Löschung ohne ausdrückliche Erlaubnis,
* keine Veränderung auftragsfremder Dateien,
* kein automatischer Git-Commit ohne ausdrückliche Erlaubnis,
* kein automatischer Reset,
* kein Überschreiben fremder uncommitteter Änderungen,
* keine Vermischung aktiver und historischer Artefakte,
* kein Ersetzen kanonischer Artefakte durch nicht kanonische Kopien,
* keine Änderung von Authentifizierung oder Berechtigungen ohne Sonderfreigabe,
* keine Annahme erfolgreicher Ausführung ohne Nachweis,
* kein Verschweigen fehlgeschlagener Tests,
* kein Erfinden nicht geprüfter Projektzustände,
* keine automatische Selbstfreigabe,
* kein Umgehen von Governance-Gates,
* keine verdeckten Nebenänderungen,
* keine unkontrollierten externen Netzwerkzugriffe,
* keine Verarbeitung von Geheimnissen außerhalb freigegebener Grenzen,
* keine Aktivierung autonomer Wartung durch diesen Auftrag.

# 22. Phase Q – Fehler-, Abbruch- und Rückfallregeln

Codex soll verbindliche Abbruchbedingungen definieren.

Mindestens:

* Scope unklar,
* Agentenidentität nicht gültig,
* Berechtigung fehlt,
* Risikoklasse nicht bestimmbar,
* Rückfallpunkt fehlt,
* unerwartete Änderungen im Arbeitsbaum,
* Konflikt mit kanonischen Regeln,
* Tests nicht ausführbar,
* Testfehler außerhalb des erwarteten Bereichs,
* unerwartete Architekturabhängigkeiten,
* kritische Sicherheitsbefunde,
* erforderliche Freigabe fehlt,
* Auftrag und tatsächlicher Projektzustand widersprechen sich.

Bei Abbruch muss der Agent:

1. keine weiteren schreibenden Aktionen ausführen,
2. den Zustand sichern,
3. bereits vorgenommene Änderungen nennen,
4. Risiken dokumentieren,
5. klar zwischen Fehler, Blockade und bewusster Ablehnung unterscheiden,
6. keine automatische Rücksetzung fremder Änderungen durchführen.

# 27. Phase V – Teststrategie

* erlaubte Aktion im erlaubten Scope,

* verbotene Aktion im erlaubten Scope,

## 27.3 Gate-Tests

* fehlende Identität,
* fehlender Auftrag,
* fehlende Capability,
* fehlende Permission,
* fehlende Freigabe,

* fehlgeschlagene Verifikation.

## 27.4 Rollen-Tests

* Analysis Agent darf nicht schreiben,
* Test Agent darf nicht freigeben,
* Implementation Agent darf Scope nicht erweitern,
* Review Agent verändert den geprüften Code nicht,
* Release Agent darf keine ungetesteten Änderungen freigeben.

* alle Aktionen werden protokolliert,

## 27.6 Integrationstests

* CRE
* Execution Planner
* Orchestrator
* Authentication
* CAM
* Release Integrity
* Audit und Provenienz

# 31. Abbruchbedingungen dieses Auftrags

Codex muss den Auftrag abbrechen oder auf reine Analyse begrenzen, wenn:

* die aktuelle Projektstruktur nicht eindeutig bestimmbar ist,
* ein bestehendes gleichartiges Framework gefunden wird,
* zentrale Governance-Artefakte widersprüchlich sind,
* ein Rückfallpunkt nicht bestimmbar ist,
* auftragsfremde uncommittete Änderungen gefährdet wären,
* die Einführung des CCAF bestehende produktive Komponenten unmittelbar verändern würde,
* eine klare Abgrenzung zu CDF, CDG, CRE, Execution Planner oder Orchestrator nicht möglich ist,
* autonome Rechte nur durch Sicherheitsabsenkung vorbereitet werden könnten,
* erforderliche Projektartefakte fehlen.

In diesem Fall ist kein improvisierter Ersatz zu implementieren.

---

# 32. Nicht zulässige Ergebnisse

Der Auftrag gilt nicht als erfolgreich abgeschlossen, wenn lediglich:

* eine allgemeine Beschreibung von Code-Agenten erstellt wird,
* bestehende Projektstrukturen nicht geprüft werden,
* Rollen ohne Berechtigungsmodell definiert werden,
* Capabilities mit Permissions gleichgesetzt werden,
* keine Risikoklassen vorgesehen werden,
* kein Vier-Augen-Prinzip berücksichtigt wird,
* keine Audit- und Provenienzstruktur vorgesehen wird,
* Codex-spezifische Regeln fälschlich als allgemeines Framework dargestellt werden,
* autonome Schreibrechte stillschweigend eingeführt werden,
* eine neue Parallelarchitektur neben bestehenden Komponenten entsteht,
* die Abgrenzung zu CDF, CDG, CRE, Execution Planner und Orchestrator unklar bleibt.

---

# 33. Leitentscheidung

Das CCAF soll nicht möglichst viele autonome Fähigkeiten schaffen.

Es soll zunächst sicherstellen, dass automatisierte Codearbeit:

* identifizierbar,
* begrenzt,
* genehmigt,
* nachvollziehbar,
* prüfbar,
* reproduzierbar,
* widerrufbar,
* architekturkonform

erfolgt.

Der zentrale Leitgedanke lautet:

> Erst müssen Identität, Auftrag, Rolle, Fähigkeit, Berechtigung, Risiko, Kontrolle und Nachweis verbindlich geregelt sein. Danach kann entschieden werden, welche Code-Agenten in welchem Umfang tatsächlich handeln dürfen.
