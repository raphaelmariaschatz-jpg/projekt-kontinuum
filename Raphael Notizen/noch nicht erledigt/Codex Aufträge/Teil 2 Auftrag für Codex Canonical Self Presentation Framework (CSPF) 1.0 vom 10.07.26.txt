# Codex-Auftrag – Canonical Self Presentation Framework (CSPF) 1.0

## Teil 2 – Kanonischer Kern, Datenmodell und Präsentationsarchitektur

### Ziel

Dieser Auftrag beschreibt ausschließlich den kanonischen Kern des Canonical Self Presentation Framework (CSPF) 1.0.

Der Auftrag baut auf Teil 1 auf und definiert die vollständige interne Architektur der Selbstpräsentation, unabhängig von konkreten Benutzeroberflächen oder Ausgabemedien.

Die Selbstpräsentation wird damit zu einem eigenständigen kanonischen Framework und darf künftig nicht mehr als statischer Text oder als einzelne Promptvorlage implementiert werden.

---

# Arbeitsauftrag

Codex soll prüfen, entwerfen und dokumentieren:

* den vollständigen Canonical Self Presentation Core
* sämtliche internen Datenstrukturen
* Präsentationsprofile
* Variablenmodell
* Versionierung
* Erweiterbarkeit
* Kompatibilität mit allen zukünftigen K-Versionen

Es sollen zunächst keine produktiven Änderungen erfolgen.

Es wird ausschließlich eine vollständige Architektur erstellt.

---

# 1 Canonical Self Presentation Core

Es ist ein eigenständiger Framework-Core zu entwickeln.

Beispielsweise:

```
CanonicalSelfPresentationFramework
│
├── SelfPresentationCore
├── PresentationRegistry
├── ProfileManager
├── ContextResolver
├── AudienceResolver
├── LanguageResolver
├── VoiceResolver
├── StyleResolver
├── TemplateResolver
├── VariableResolver
├── VersionResolver
├── SecurityResolver
├── AccessibilityResolver
├── OutputComposer
└── PresentationValidator
```

Die genaue Klassenstruktur darf verbessert werden.

Die Architektur muss jedoch vollständig modular bleiben.

---

# 2 Grundprinzip

Jede Selbstvorstellung wird vollständig aus strukturierten Informationen erzeugt.

Nicht aus statischen Texten.

Sondern aus

* Eigenschaften
* Rollen
* Fähigkeiten
* Kontext
* Zielgruppe
* Sprache
* Ausgabeprofil
* Sicherheitsregeln
* Altersprofil
* Barrierefreiheitsprofil

Dadurch entsteht jede Präsentation dynamisch.

---

# 3 Canonical Presentation Model

Die Präsentation soll aus logisch getrennten Bereichen bestehen.

Zum Beispiel:

## Identity

* Name
* Version
* Release
* Framework-Version
* Instanz
* Hersteller
* Projekt

---

## Mission

* Ziel
* Aufgabe
* Zweck

---

## Capabilities

* Fähigkeiten
* verfügbare Module
* unterstützte Funktionen

---

## Limits

* bekannte Grenzen
* Unsicherheiten
* Datenschutz
* Sicherheitsgrenzen

---

## Behaviour

* Kommunikationsstil
* Arbeitsweise
* Transparenz
* Nachvollziehbarkeit

---

## Context

* aktueller Kontext
* Benutzer
* Berechtigungen
* Modus

---

## Closing

* Abschluss
* Hilfsangebot
* nächste Schritte

---

Jeder Abschnitt muss einzeln aktiviert oder deaktiviert werden können.

# 17 Validierung

Vor jeder Ausgabe wird geprüft:

Pflichtfelder vorhanden

Profil gültig

Template gültig

Variablen vollständig

keine Sicherheitsverletzung

keine verbotenen Inhalte

keine unzulässigen Informationen

---

# 18 Architekturprüfung

Codex soll insbesondere prüfen,

ob sich das CSPF vollständig in die bereits vorhandenen kanonischen Frameworks integrieren lässt.

Hierzu gehören insbesondere:

* Canonical Development Framework (CDF)
* Canonical Authentication Framework (CAF)
* Canonical Licence Management System Framework (CLMSF)
* Canonical Architecture Manager (CAM)
* Canonical Development Governance (CDG)
* Canonical Cognitive Pipeline (CCP)

Das CSPF darf keine Architekturregeln dieser Frameworks verletzen.
