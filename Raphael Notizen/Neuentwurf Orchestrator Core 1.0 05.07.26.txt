Codex-Auftrag – Orchestrator Core 1.0 neu entwerfen und vorbereiten – 05.07.26

Ziel:
Entwerfe den Orchestrator Core 1.0 als neue zentrale Execution Runtime von Projekt Kontinuum.

Wichtig:
Nicht den bisherigen Orchestrator weiterbauen.
Nicht den alten direkten Agentenpfad ausbauen.
Der Orchestrator Core 1.0 soll auf der neuen Architektur nach CRE 1.0 und Execution Planner 1.0 basieren.

Grundprinzip:
Der Orchestrator Core 1.0 übernimmt ausschließlich validierte ExecutionPlans vom Execution Planner und verwaltet deren Ausführung.

Er entscheidet keine Capabilities.
Er erzeugt keine Pläne.
Er ersetzt nicht CRE.
Er ersetzt nicht den Execution Planner.

3. ExecutionPlan übernehmen

Der Orchestrator muss:

* einen ExecutionPlan empfangen
* plan_id und request_id prüfen
* status prüfen
* steps prüfen
* parallel_groups prüfen
* validation prüfen
* blockierte, leere oder ungültige Pläne ablehnen

Keine eigene Planerzeugung.

4. Agent Lifecycle verwalten

Für jeden Schritt aus dem Plan:

* expected_agent auslesen
* Agentenverfügbarkeit prüfen
* Agent starten
* Ergebnis entgegennehmen
* handled=True / handled=False auswerten
* Fehler erfassen
* Laufstatus dokumentieren

5. Fehlerbehandlung

Behandle mindestens:

* fehlender Agent
* Agent nicht verfügbar
* Agent antwortet nicht
* handled=False
* ungültiges Ergebnis
* Timeout
* Governance-Blockierung
* ungültiger Plan
* leerer Plan

6. Fallback-Steuerung

Wenn handled=False oder Agent nicht verfügbar:

* Fallback nur kontrolliert versuchen
* keine freie Agentenwahl außerhalb des Plans
* Fallback muss dokumentiert werden
* Fallback muss Governance respektieren
* wenn kein Fallback erlaubt ist: sauber blockieren

7. Parallelgruppen vorbereiten

Der Orchestrator soll parallel_groups aus dem ExecutionPlan lesen.

Version 1.0 darf noch sequentiell ausführen, muss aber:

* Gruppenstruktur erkennen
* Schritte gruppenweise abarbeiten
* spätere Parallelisierung ermöglichen

8. ExecutionRun erzeugen

Jede Ausführung erzeugt einen ExecutionRun mit mindestens:

* run_id
* plan_id
* request_id
* started_at
* finished_at
* status
* tasks
* results
* errors
* fallbacks
* governance_decisions
* review_handoff_ready
* cmm_handoff_ready

9. Governance respektieren

Vor Ausführung prüfen:

* Plan validation ok
* Governance-Level
* blockierte Schritte
* execution_allowed falls vorhanden

Governance darf nicht umgangen werden.

10. Review- und CMM-Handoff vorbereiten

Der Orchestrator führt Review und CMM noch nicht selbst aus.

Er soll nur vorbereiten:

* review_handoff_ready=True bei erfolgreicher Ausführung
* cmm_handoff_ready=True erst nach Review-Freigabe vorbereitet, aber nicht dauerhaft speichern
