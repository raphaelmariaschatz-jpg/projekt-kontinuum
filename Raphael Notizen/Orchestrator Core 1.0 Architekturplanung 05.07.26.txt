Orchestrator Core 1.0 – Architekturplanung

Ziel:
Der Orchestrator Core 1.0 ist die zentrale Execution Runtime von Projekt Kontinuum.

Er entscheidet nicht, welche Capability benötigt wird.
Er plant nicht selbst.
Er führt keine freie Agentensuche durch.

Er übernimmt ausschließlich einen validierten ExecutionPlan vom Execution Planner und verwaltet dessen Ausführung.

Kanonische Rolle:

CRE
→ erkennt Capabilities, Prioritäten, Governance-Hinweise

Execution Planner
→ erzeugt deterministischen ExecutionPlan

Orchestrator Core
→ führt validierte ExecutionPlans kontrolliert aus

Governance
→ erlaubt, blockiert oder eskaliert Ausführung

Agent
→ erledigt fachliche Aufgabe

Review
→ prüft Ergebnis

CMM
→ speichert kanonisch

Aufgaben des Orchestrator Core 1.0:

1. ExecutionPlan übernehmen

* Plan empfangen
* Planstruktur prüfen
* Status prüfen
* Governance-Level prüfen
* leere, blockierte oder ungültige Pläne ablehnen

2. Agent Lifecycle verwalten

* passenden Agenten aus dem Plan lesen
* Agentenverfügbarkeit prüfen
* Agent starten
* Ergebnis entgegennehmen
* handled=True / handled=False auswerten
* Laufstatus dokumentieren

3. Fehlerbehandlung

* fehlender Agent
* Agent antwortet nicht
* Agent gibt Fehler zurück
* ungültiges Ergebnis
* Timeout
* blockierte Governance
* unvollständiger Plan

4. Fallback-Steuerung

* geeigneten Fallback-Agenten suchen
* Fallback nur nach Governance-Regeln erlauben
* Fallback dokumentieren
* keine unkontrollierte freie Agentenwahl

5. Parallelisierung vorbereiten

* Parallelgruppen aus ExecutionPlan lesen
* Gruppen nacheinander ausführen
* Schritte innerhalb einer Gruppe später parallelisierbar machen
* In Version 1.0 darf die Ausführung noch sequentiell simuliert werden, aber die Struktur muss parallelisierbar sein

6. Runtime-Status erzeugen
   Jede Ausführung erzeugt einen ExecutionRun mit:

* run_id
* plan_id
* request_id
* gestartete Schritte
* Agentenstatus
* Ergebnisse
* Fehler
* Fallbacks
* Governance-Entscheidungen
* Gesamtstatus

7. Review-Handoff vorbereiten

* Nach erfolgreicher Ausführung Ergebnis an Review-Schicht übergeben
* Noch keine Review-Logik selbst implementieren

8. CMM-Handoff vorbereiten

* Nur geprüfte Ergebnisse dürfen später kanonisch gespeichert werden
* Orchestrator speichert nicht selbst dauerhaft im CMM
* Er erzeugt nur den Übergabestatus

Nicht-Aufgaben:

* keine Capability-Erkennung
* keine Planung
* keine CRE-Logik
* keine direkte Wissensantwort
* keine freie Agentenentscheidung
* keine Memory-Speicherung
* keine Review-Bewertung
* keine Geschäftslogik

* 17_tests/test_orchestrator_core_1_0.py

Tests:

* gültiger ExecutionPlan wird angenommen
* blockierter Plan wird abgelehnt
* fehlender Agent wird erkannt
* Agent wird gestartet
* Ergebnis wird empfangen
* handled=True wird korrekt gesetzt
* handled=False löst Fallback aus
* Fallback wird dokumentiert
* Timeout wird behandelt
* Governance-Blockierung verhindert Ausführung
* Parallelgruppen werden korrekt gelesen
* Orchestrator erzeugt ExecutionRun
* Orchestrator plant nicht selbst
* Orchestrator ruft CRE nicht direkt auf
* Orchestrator führt keine ungeplanten Agenten aus

Zielzustand:
Der Orchestrator Core 1.0 ersetzt den alten direkten Runtime-Pfad durch eine kontrollierte Ausführungslogik auf Basis validierter ExecutionPlans.

Umsetzung:

Verstanden. Das ist jetzt nicht mehr nur Prüfung, sondern der Architekturauftrag für Orchestrator Core 1.0 als reine Execution Runtime. Ich lese zuerst den aktuellen Planner-, Agent- und Runtime-Stand, damit der neue Core sauber an ExecutionPlan anschließt und den alten Direktpfad nicht blind kopiert.

Ich habe die Agent-Schnittstelle gefunden: Agenten liefern AgentResult(agent, handled, answer). Der Orchestrator-Core kann also minimal und sauber ohne Kenntnis einzelner Fachagenten arbeiten: Registry der vorhandenen Agentobjekte, Step-Payload rein, Ergebnis/Fehler/Fallback raus.

Ich setze jetzt die drei Kernartefakte auf. Dazu nehme ich bewusst keine CRE-/Planner-Abhängigkeit in den Orchestrator auf, sondern nur ExecutionPlan als Dict und eine Agent-Registry; die Tests werden genau diese Grenze absichern.
