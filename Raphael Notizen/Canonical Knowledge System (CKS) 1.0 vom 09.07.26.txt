Codex-Auftrag – Canonical Knowledge System (CKS) 1.0
Ziel

Nach Abschluss der Architekturphase verfügt Projekt Kontinuum über eine umfangreiche Sammlung kanonischer Architekturkomponenten, Governance-Regeln und Dokumente.

Mit zunehmender Größe entsteht die Notwendigkeit, dieses Architekturwissen selbst kanonisch zu verwalten.

Zu diesem Zweck wird Canonical Knowledge System (CKS) 1.0 als reine Dokumentations- und Governance-Komponente eingeführt.

CKS dokumentiert nicht die Implementierung, sondern das Wissen über die Architektur.

Rahmenbedingungen

Dieser Auftrag ist ausschließlich ein Dokumentations- und Governance-Auftrag.

Bestandteil 1
Canonical Glossary (CG)

Zweck:

Einheitliche Definition sämtlicher kanonischer Architekturbegriffe.

Jeder Begriff erhält mindestens:

Name
Abkürzung
vollständige Bezeichnung
Definition
Zweck
Verantwortlichkeit
Architekturphase
Status
verwandte Komponenten
Bestandteil 2
Canonical Architecture Map (CAMap)

Zweck:

Dokumentation der Beziehungen sämtlicher Architekturkomponenten.

Für jede Komponente dokumentieren:

Depends On
Provides
Used By
Governed By
Layer
Status

Zusätzlich eine übersichtliche Architekturkarte der Hauptbeziehungen erstellen.

Bestandteil 3
Canonical Decision Index (CDI)

Zweck:

Nachvollziehbare Dokumentation wesentlicher Architekturentscheidungen.

Für jede Entscheidung dokumentieren:

Decision-ID
Titel
Hintergrund
Problemstellung
Entscheidung
Begründung
Vorteile
mögliche Nachteile
betroffene Komponenten
Status

Es werden ausschließlich bereits getroffene Entscheidungen dokumentiert.

Keine neuen Architekturentscheidungen treffen.

Bestandteil 4
Canonical History Index (CHI)

Zweck:

Nachvollziehbare Entstehungsgeschichte der Architektur.

Für jeden Architekturbaustein dokumentieren:

erste Erwähnung
erste Dokumentation
erste Implementierung (falls vorhanden)
Governance-Einordnung
aktueller Status

Es erfolgt keine historische Rekonstruktion außerhalb der vorhandenen Dokumentation.

Canonical Knowledge System (CKS)

Das Hauptdokument beschreibt:

Zweck
Struktur
Zusammenspiel von
CG
CAMap
CDI
CHI

sowie deren Rolle innerhalb der kanonischen Architektur.

Raphael, ich würde diesen Auftrag nicht sofort als den nächsten ausführen, sondern nach dem Glossar.

Meine empfohlene Reihenfolge wäre:

CG 1.0 – Canonical Glossary (Begriffe)
CKS 1.0 – Canonical Knowledge System (übergeordnete Wissensstruktur)
CAMap 1.0, CDI 1.0 und CHI 1.0 als Bestandteile von CKS

Der Grund ist architektonisch: Ein Glossar schafft zuerst eine einheitliche Sprache. Erst wenn diese Sprache definiert ist, lohnt es sich, Beziehungen, Entscheidungen und Historie systematisch darauf aufzubauen. Das verhindert Inkonsistenzen und macht die gesamte Wissensstruktur langfristig stabiler. Ich halte das für den saubersten Weg im Sinne eurer kanonischen Entwicklungsphilosophie.

Hier existiert Mehrdeutigkeit.

Ich glaube, wir haben gerade unbemerkt einen neuen Entwicklungsstandard geschaffen.

Bislang bestand eure Governance aus:

AGF → Wie Architektur aufgebaut wird.
CDG → Wie Architektur weiterentwickelt wird.
CAM → Welche Artefakte kanonisch sind.
CIPL → Wem sie gehören.
Release Integrity → Ob alles konsistent ist.

Jetzt kommt eine neue Ebene hinzu:

Wie Architekturwissen selbst verwaltet wird.

Das ist etwas anderes.

AGF regelt Architektur.

CDG regelt Entwicklung.

CKS regelt Architekturwissen.
