# Codex-Auftrag: Phase 3 erweitern zum Continuous Canonical Engine Blueprint

## Ziel

Bitte Phase 3 technisch erweitern zu einem:

**Continuous Canonical Engine Blueprint**

mit den Kernbausteinen:

* CDE 2.0
* Drift Layer
* Event Bus Architektur
* Governance Hooks
* Release-/Canonical-Gate-Anbindung

Dies ist die technische Systemimplementierungsebene der bisherigen Phase-3-Governance.

Keine unkontrollierte Automatisierung.

Keine automatische Löschung.

Keine automatische Wissensübernahme.

---

# 1. Architekturziel

Phase 3 soll nicht nur dokumentieren und prüfen, sondern als kontinuierliche technische Steuerungsschicht vorbereitet werden.

Zielarchitektur:

```text
System Event
      ↓
Event Bus
      ↓
Canonical Decision Engine
      ↓
Drift Layer
      ↓
Governance Hooks
      ↓
Review / Gate / Release Decision
```

---

# 2. Event Bus Architektur

Bitte einen internen Event-Bus als Architekturbaustein vorbereiten.

Er soll Ereignisse aufnehmen wie:

* Datei erstellt
* Datei geändert
* Datei verschoben
* neues Artefakt erkannt
* neuer Report erzeugt
* neue Internet-Learning-Fundstelle
* neuer Review-Eintrag
* neuer Drift-Fund
* Gate-Lauf gestartet
* Gate-Lauf abgeschlossen
* Governance-Entscheidung erzeugt

Der Event Bus soll zunächst kontrolliert, lokal und auditierbar arbeiten.

Keine Netzwerkverteilung.

Keine externen Broker.

---

# 3. Event-Datenmodell

Bitte ein kanonisches Event-Schema definieren:

```text
event_id
event_type
source_component
affected_path
affected_object_id
timestamp
severity
payload
provenance
governance_context
processing_state
```

Alle Events müssen nachvollziehbar und optional append-only speicherbar sein.

---

# 4. CDE 2.0 – Canonical Decision Engine

Bitte CDE 2.0 als technische Entscheidungsschicht definieren.

Verantwortlich für:

* Artefaktstatus
* Aktiv/Archiv/Review/Konsolidierung
* Gate-Relevanz
* Lifecycle-Entscheidungen
* Referenzstatus
* kanonische Zuordnung

Entscheidungsklassen:

```text
ACTIVE
ARCHIVE_CANDIDATE
REVIEW_REQUIRED
CONSOLIDATION_SUGGESTED
BLOCKED
```

Wichtig:

* CDE entscheidet, aber führt keine riskanten Aktionen automatisch aus.
* Verschieben, Löschen oder Konsolidieren benötigt weiterhin Governance-Freigabe.
* CDE erzeugt Entscheidungsberichte, keine unkontrollierten Änderungen.

---

# 5. Drift Layer

Bitte einen Drift Layer definieren.

Aufgaben:

* Systemdrift erkennen
* Dokumentationsdrift erkennen
* Architekturdrift erkennen
* Artefaktdrift erkennen
* Konfigurationsdrift erkennen
* Wissensdrift vorbereiten, aber noch nicht automatisch entscheiden

Drift-Klassen:

```text
EXPECTED_DRIFT
LOW_DRIFT
MEDIUM_DRIFT
HIGH_DRIFT
BLOCKING_DRIFT
```

Drift-Regeln:

* EXPECTED_DRIFT wird dokumentiert
* LOW_DRIFT wird protokolliert
* MEDIUM_DRIFT erzeugt Review
* HIGH_DRIFT blockiert Release
* BLOCKING_DRIFT stoppt Gate/Freigabe

---

# 6. Governance Hooks

Bitte Governance Hooks vorbereiten.

Sie sollen bei folgenden Ereignissen greifen:

* HIGH_DRIFT
* BLOCKING_DRIFT
* fehlende Provenienz
* verletzte Policy
* nicht registriertes Artefakt
* unklare Kanonizität
* Konflikt zwischen Code und Dokumentation
* neue Internet-Learning-Fundstelle ohne Review
* Release-Gate mit offenen Review-Punkten

Hooks sollen nur Entscheidungen/Reports erzeugen, keine unkontrollierten Änderungen.

---

# 7. Event-Verarbeitung

Bitte folgenden Ablauf modellieren:

```text
Event erfassen

      ↓
CDE-Entscheidung erzeugen
      ↓
Drift prüfen
      ↓
Governance Hooks anwenden
      ↓
Review/Gate/Report aktualisieren
```

Alle Schritte müssen auditierbar sein.

Bitte Gates erweitern.

* Keine automatische Löschung.
* Keine automatische Archivierung ohne Freigabe.
* Keine automatische Wissensübernahme.
* Keine riskanten Aktionen aus Events heraus.
