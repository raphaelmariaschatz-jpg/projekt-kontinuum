Codex-Auftrag: CodeAgent für Projekt Kontinuum integrieren

Ziel:
Bitte einen neuen CodeAgenten in Projekt Kontinuum integrieren, der Quellcode analysieren, Projektstrukturen verstehen, Programmierlogik erklären und perspektivisch neue Programmiersprachen lernen kann.

Grundsatz:
Der CodeAgent arbeitet zunächst read-only. Er darf Quellcode lesen, analysieren und Berichte erstellen, aber keine Quelldateien verändern, verschieben oder automatisch reparieren.

Aufgaben:

1. Bestehende Architektur prüfen

* vorhandene Agentenstruktur analysieren
* bestehende Registry-, Router-, Memory-, Knowledge- und Provenienzlogik verwenden
* keine Parallelarchitektur erzeugen
* CodeAgent sauber in die vorhandene Agent Registry eintragen

2. CodeAgent als eigenes Modul erstellen
   Vorgeschlagener Name:

* CodeAgent

Zentrale Fähigkeiten:

* Quellcodedateien lesen
* Projektordner analysieren
* Programmiersprache erkennen
* Dateien nach Sprache, Zweck und Risiko klassifizieren
* Imports, Abhängigkeiten und Einstiegspunkte erkennen
* Klassen, Funktionen, Methoden und zentrale Logik extrahieren
* einfache Architekturzusammenfassung erstellen
* bekannte Risiken, TODOs, fehlende Tests und auffällige Strukturen melden

3. Unterstützte Dateitypen
   Mindestens vorbereiten für:

* Python: .py
* JavaScript/TypeScript: .js, .jsx, .ts, .tsx
* Web: .html, .css
* Shell/Batch: .sh, .bat, .ps1
* Konfiguration: .json, .yaml, .yml, .toml, .ini, .env.example
* Dokumentation: .md, .txt

Originaldateien dürfen nicht verändert werden.

4. Zentrale Methoden vorbereiten

Bitte mindestens folgende Methoden umsetzen:

analyze_file(path: str) -> dict

analyze_project(path: str) -> dict

detect_language(path: str, content: str) -> dict

summarize_code(path: str, content: str) -> dict

extract_symbols(path: str, content: str) -> dict

create_project_map(path: str) -> dict

Rückgaben bitte als strukturierte Dict-/JSON-Blöcke mit:

* status
* source_path
* language
* file_type
* detected_symbols
* imports/dependencies
* entrypoints
* summary
* risks
* warnings
* errors
* timestamp
* agent_name
* read_only: true

5. Router-Anbindung
   Neue Befehle ermöglichen, z. B.:

* "analysiere code <pfad>"
* "analysiere projekt <pfad>"
* "erkläre code <pfad>"
* "finde einstiegspunkte <pfad>"
* "erstelle projektkarte <pfad>"
* "welche sprache ist <pfad>"
* "codeagent <pfad>"

Der Request Router soll solche Anfragen an den CodeAgent weiterleiten.

6. Projektverständnis
   Der CodeAgent soll bei Ordneranalysen erkennen:

* Hauptsprache des Projekts
* wichtige Einstiegspunkte
* zentrale Module
* Testordner
* Konfigurationsdateien
* Dokumentationsdateien
* mögliche Build-/Startbefehle
* grobe Architektur
* auffällige Altdateien, Duplikate oder unklare Strukturen

7. Lernfähigkeit für neue Programmiersprachen vorbereiten
   Bitte eine erweiterbare Sprachregistry anlegen oder nutzen, z. B.:

language_registry.json

Darin sollen pro Sprache gespeichert werden können:

* Dateiendungen
* Kommentarzeichen
* typische Import-/Include-Muster
* Funktions-/Klassenmuster
* Einstiegspunkt-Konventionen
* Paket-/Build-Dateien
* bekannte Risiken
* empfohlene Analyseparser

Wichtig:
Neue Sprachen sollen später durch Konfiguration ergänzt werden können, ohne den CodeAgent selbst hart umzubauen.

8. Knowledge-/Memory-Anbindung vorbereiten
   Der CodeAgent soll Analyseergebnisse optional später in Memory oder Knowledge Graph speichern können.

Jetzt aber:

* keine automatische Speicherung ohne ausdrücklichen Befehl
* klare Trennung zwischen Analyse und Persistenz
* exportierbare JSON-Ergebnisse
* vollständige Provenienzangaben

9. Sicherheit und Pfadfreigabe
   Der CodeAgent muss bestehende Pfadfreigabe- und Sicherheitsregeln respektieren.

Bei nicht freigegebenem Pfad:

* sauber ablehnen
* verständliche Fehlermeldung ausgeben
* keine Datei öffnen

Bei sehr großen Projekten:

* Größenlimit beachten
* Analyse abbrechen oder begrenzen
* Teilergebnis mit Warnung zurückgeben

10. GUI-Vorbereitung
    Falls die GUI Agentenstatus oder Quellenausgabe unterstützt:

* CodeAgent dort sichtbar machen
* Quelle "lokaler Quellcode" anzeigen
* erkannte Sprache anzeigen
* Read-only-Status anzeigen
* Risiken/Warnungen verständlich ausgeben

11. Tests erstellen
    Bitte Tests für folgende Fälle ergänzen:

* gültige Python-Datei
* gültige JavaScript-Datei
* Projektordner mit mehreren Dateien
* nicht vorhandene Datei
* nicht unterstützter Dateityp
* nicht freigegebener Pfad
* sehr große Datei
* Router-Befehl "analysiere code <pfad>"
* Router-Befehl "analysiere projekt <pfad>"

12. Abschlussbericht erstellen
    Bitte nach Umsetzung einen kurzen Bericht erzeugen mit:

* geänderten Dateien
* neu angelegten Dateien
* Registry-/Router-Anbindung
* unterstützten Sprachen und Dateitypen
* Testergebnissen
* bekannten Einschränkungen
* nächstem sinnvollen Ausbauschritt

Wichtig:
Keine Quelldateien verändern.
Keine automatische Reparatur.
Keine automatische Speicherung ins Memory.
Keine Parallelarchitektur.
Read-only zuerst.
Kanonische Integration vor Funktionsfülle.
